# 内投反馈-先投后处刑

依据案例：batch-260 案例 1296

## 读者承诺

公开处刑能提供建议，但更高性价比的流程是先投稿或内投，用编辑反馈确定问题方向，再回到社群求精修。作者不应把同一开头反复处刑到失去判断。

## 核心循环

```text
完成开头 -> 先投递 -> 获得硬反馈 -> 带反馈处刑 -> 定向修改
```

## 开篇必备

- 稿件当前版本。
- 是否已投递。
- 编辑是否反馈。
- 本次处刑只问什么。
- 哪些建议不纳入本轮修改。

## 期待来源

作者期待从“泛泛求评价”转成“针对过稿障碍解决问题”。有了编辑反馈，公开处刑就不再是随机意见收集。

## 断档风险

如果没投就反复处刑，容易被小方向意见带偏；如果大方向错却只做细节修补，也会浪费时间。

## 训练任务

为一份开头写投递优先级表：先投、等反馈、再处刑、改一版、停止修改。每一步限定触发条件。
