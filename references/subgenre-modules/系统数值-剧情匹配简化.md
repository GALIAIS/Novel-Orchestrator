# 系统数值-剧情匹配简化

依据案例：`batch-065` 案例 324。

## 读者承诺

这个模块用于系统文、任务流、签到流、宝箱流。它承诺的是不把系统写成精算表，而让数值和奖励粗粒度服务剧情：当前问题需要什么，系统就给什么层级的反馈。

## 核心循环

```text
剧情问题 -> 系统任务 -> 粗粒度奖励 -> 解决当前问题 -> 新功能/新宝箱拉期待
```

## 开篇必备

- 系统功能要服务当前剧情。
- 不细写复杂积分兑换比例。
- 奖励效果比奖励数值重要。
- 签到、任务、宝箱优先于复杂数值表。
- 系统不能喧宾夺主。

## 期待来源

- 高频收获带来爽点密度。
- 任务完成后立刻有用。
- 宝箱或签到带来下一次期待。
- 奖励解决当前困境。
- 新功能逐步展开。

## 断档风险

- 数值太细，作者算得累。
- 读者觉得水字数。
- 较真读者挑数值漏洞。
- 抽奖奖品库拖垮作者。
- 系统盖过人物和剧情。

## 训练任务

把一个积分兑换系统改成宝箱系统。每次只写任务、完成动作、奖励效果和下一次期待，不写兑换比例。
