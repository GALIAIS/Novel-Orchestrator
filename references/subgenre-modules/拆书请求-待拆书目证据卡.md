# 拆书请求-待拆书目证据卡

依据案例：batch-292 案例 1460

## 读者承诺

只有拆书请求、没有正文和评论时，只记录待拆入口，不硬编拆书结论。

## 核心循环

```text
点名请求 -> 记录书名 -> 标注题材方向 -> 列证据缺口 -> 等样本补拆
```

## 开篇必备

- 待拆书名。
- 平台。
- 疑似题材。
- 提名原因。
- 当前材料。
- 缺失材料。
- 后续拆解问题。

## 期待来源

系统保留高频火书入口，后续拿到章节样本后能快速进入精拆。

## 断档风险

- 把请求当成果。
- 无材料硬拆。
- 靠书名联想污染案例库。
- 待拆书名没有索引，后续丢失。

## 训练任务

为一本待拆火书建卡：书名、平台、题材、现有材料、缺失材料、优先拆哪里。
