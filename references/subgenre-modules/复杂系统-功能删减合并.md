# 复杂系统-功能删减合并

依据案例：`batch-172` 案例 859。

## 读者承诺

手游式系统可以借用抽卡、任务、等级、培养，但开头必须删减合并功能，让每次行动对应明确收获。

## 核心循环

```text
功能堆叠 -> 删除重复 -> 延后解锁 -> 任务分发奖励 -> 等级控制节奏
```

## 开篇必备

- 第一章只展示一个核心系统功能。
- 抽卡、盲盒、签到、商城功能重叠时要合并。
- 奖励来源要和主角行动绑定。
- 等级任务负责控制主角成长节奏。

## 期待来源

读者期待做事有回报、升级有变化、系统规则容易理解。功能少而清楚，比功能多而混乱更能追读。

## 断档风险

- 一口气抛出太多系统 UI。
- 奖励全靠作者说了算，没有因果。
- 签到导致战力膨胀。
- 系统功能服务展示，不服务主线。

## 训练任务

把一个复杂系统开头删到 3 个核心功能。第一章只出现主功能，其余功能写明解锁等级和对应剧情用途。
