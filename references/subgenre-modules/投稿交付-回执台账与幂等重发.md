# 投稿交付-回执台账与幂等重发

依据案例：`batch-398` 案例 1989

## 读者承诺

把邮件投稿变成可追踪交付：标准化投稿包、记录版本与承诺时间、区分未送达和未通过，并在要求重发时保持唯一有效正文。

## 核心循环

```text
标准投稿包 -> 发送并记账 -> 回执或承诺窗口 -> 状态确认 -> 同版本幂等重发 -> 有限跟进或转官方备用路径
```

## 开篇必备

- 书名、题材、简介、样章和联系方式齐全。
- 文件名和正文有明确版本号。
- 记录邮箱、时间、主题和附件。
- 回执与预计处理时间进入台账。
- 重发说明原日期和编辑指示。
- 内容更新时声明唯一有效版本。

## 期待来源

作者能确认稿件当前处于送达、审阅、补件、通过或拒绝中的哪一状态，不把邮件故障误读成作品评价。

## 断档风险

- 把已发送当成已进入审阅。
- 无回复就无证据改稿。
- 多邮箱发送不同版本。
- 重发时悄悄替换正文。
- 高频催促造成沟通噪声。
- 没有退出和转投条件。

## 训练任务

建立投稿模板与状态台账，模拟首次发送、无回执、确认、同版本重发和最终结果；任何内容变化都必须产生新版本号并指定唯一有效文件。
