# 系统流-任务反馈结构

依据案例：`batch-008` 案例 036。

## 读者承诺

系统负责把主角欲望转成可执行任务，再用奖励反馈制造下一轮行动理由。读者看的不是系统面板，而是任务如何让主角持续行动、碰撞阻碍、获得成长。

## 核心循环

```text
主角欲望 -> 系统建目标 -> 主角执行 -> 阻碍升级 -> 系统反馈奖励 -> 新欲望
```

## 开篇必备

- 主角先有可理解的欲望或困境。
- 系统任务必须和当前困境直接相关。
- 第一次奖励要能改变主角处境。
- 任务不能替主角完成行动，只能逼出行动。

## 期待来源

- 读者等着看主角如何完成任务。
- 任务奖励提前吊出下一步爽点。
- 阻碍让任务不是签到领奖，而是故事事件。
- 每次奖励都扩大主角可操作空间。

## 断档风险

- 系统替主角解决全部问题，主角沦为领奖机器。
- 任务和主线欲望脱节。
- 奖励只堆数值，不改变故事局面。
- 反套路坑主角却没有价值转化，读者只感到憋屈。

## 训练任务

给一个系统写三轮任务反馈。每轮必须写清楚：主角原欲望、系统目标、阻碍、完成方式、奖励、奖励如何引出下一轮。
