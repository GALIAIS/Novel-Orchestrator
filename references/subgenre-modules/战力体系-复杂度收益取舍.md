# 战力体系-复杂度收益取舍

依据案例：`batch-084` 案例 416。

## 读者承诺

这个模块用于判断战力体系要做多复杂。它承诺让作者先评估维护成本和实际收益，避免新人硬做诡秘式复杂序列。

## 核心循环

```text
题材需求 -> 复杂度评估 -> 作者维护能力 -> 简化分级 -> 战斗兑现
```

## 开篇必备

- 明确读者是否主要追设定。
- 先选有限制或无限制。
- 控制等级层数。
- 预留能力变化来源。
- 不做维护不了的复杂体系。

## 期待来源

- 战力差距清楚。
- 能力变化不死板。
- 战斗能展开剧情。
- 复杂度服务探秘或爽点。
- 体系不拖累正文。

## 断档风险

- 为真实感堆复杂设定。
- 序列写不好反而束缚剧情。
- 等级越复杂越容易崩。
- 只满足设定党，普通读者无感。
- 新人知识量不足还硬写。

## 训练任务

同一作品写两版战力：十进制简表和复杂序列表。各写一场战斗，选择更容易产生爽点且不崩的一版。
