# 飞卢数据-上架验证优先

依据案例：`batch-058` 案例 289。

## 读者承诺

这个模块用于飞卢新书前期数据判断。它承诺的是把评价票、鲜花、收藏等前期数据当弱信号，最终仍以上架订阅验证为准。

## 核心循环

```text
前期更新 -> 记录弱数据 -> 判断是否有人看 -> 不过早下结论 -> 上架验证 -> 决定继续/调整/切书
```

## 开篇必备

- 记录当前字数和更新天数。
- 评价票只判断是否有人看。
- 不用单一数据决定生死。
- 上架前只做方向观察。
- 上架后用付费转化复盘前期信号。

## 期待来源

- 作者不被单一数据影响心态。
- 前期数据服务更新节奏调整。
- 上架节点给出真实验证。
- 复盘能积累下一本判断经验。
- 数据和内容修改分开处理。

## 断档风险

- 评价票少就立刻切书。
- 评价票多就误以为一定能成。
- 忽略上架付费转化。
- 同时看太多数据，反而失去判断。
- 用数据焦虑替代更新和内容改进。

## 训练任务

做一张 7 天飞卢数据表，只填字数、更新、收藏、鲜花、评价票、评论。第 8 天写一句“是否有人看”，不上架前不写成败结论。
