# 系统托管-主角不能废

依据案例：`batch-171` 案例 851。

## 读者承诺

系统可以强，主角可以弱，但主角必须有主动目标、选择权和不可替代价值，否则读者会把代入和期待转移到系统身上。

## 核心循环

```text
系统代办 -> 主角承担后果 -> 主角做选择 -> 系统受限制 -> 主角成长保位
```

## 开篇必备

- 系统能力要明确，但不能完全替代主角。
- 主角弱点可以写，废和拖后腿要谨慎。
- 第一次托管后要留下主角必须处理的后果。
- 系统独立人格越强，越要压住喧宾夺主风险。

## 期待来源

读者期待系统制造爽点，也期待主角配得上故事中心位置。主角承担后果和做选择，是托管流的保命线。

## 断档风险

- 系统解决所有问题，主角变成寄生壳。
- 主角好逸恶劳，读者厌恶代入对象。
- 系统人格比主角更有魅力，喧宾夺主。
- 中后期战力崩坏或任务套娃水文。

## 训练任务

写一个系统托管开头，系统只允许代办一次。代办后制造一个系统不能直接解决的问题，由主角亲自选择并承担代价。
