# 读者反馈-抬杠噪音降权

依据案例：`batch-128` 案例 636。

## 读者承诺

扫榜看反馈时，作者要区分真实毒点、阅读疑问和玩梗抬杠。高赞评论不一定代表需要修文；只有影响理解、代入和核心爽点的反馈，才值得进入修改清单。

## 核心循环

```text
收集反馈 -> 判断反馈类型 -> 识别是否影响爽点 -> 真问题修补 -> 抬杠噪音降权
```

## 开篇必备

- 先看反馈是否影响理解。
- 再看是否破坏核心爽点。
- 高赞抬杠只记录，不立刻改文。
- 奇幻/穿越设定不必被现实规则完全绑架。
- 真实毒点才进入正文修补。

## 期待来源

- 作者不被噪音拖走。
- 修文优先级更清楚。
- 读者反馈仍被尊重，但被分层处理。
- 作品核心卖点不被少数段评绑架。

## 断档风险

- 把所有高赞评论都当真。
- 为解释抬杠牺牲节奏。
- 无视真正影响代入的漏洞。
- 让现实规则压过类型爽点。

## 训练任务

找 10 条同题材段评，分成真实毒点、阅读疑问、玩梗抬杠。只给前两类写修复方案，第三类只记录为互动噪音。
