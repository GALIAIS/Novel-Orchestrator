# 同人设定-故事化补完

依据案例：`batch-077` 案例 381。

## 读者承诺

这个模块用于同人文补充原作设定。它承诺设定补完不能变成说明书，要放进剧情、爽点、战斗或角色选择里，让读者通过故事接收。

## 核心循环

```text
原作设定 -> 补充需求 -> 放入冲突 -> 行动展示差异 -> 爽点/剧情推进
```

## 开篇必备

- 不 OOC。
- 补设定必须服务当前事件。
- 设定不能大段单独讲。
- 最好通过战斗、任务或选择展示。
- 读者跳过设定段也不应失去剧情动力。

## 期待来源

- 原作熟悉感。
- 新补完带来的“原来如此”。
- 设定改变战斗或选择结果。
- 角色保持原味但事件更丰富。
- 同人补完服务原作未写之处。

## 断档风险

- 作者觉得有趣，读者觉得水。
- 设定推动取代剧情推动。
- 大量补充导致节奏停滞。
- 补设定改变角色原味。
- 把同好耐心当作无限。

## 训练任务

把一段设定说明改成一场事件：写出设定、触发冲突、角色行动、结果差异和爽点落点。
